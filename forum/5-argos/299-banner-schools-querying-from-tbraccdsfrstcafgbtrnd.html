Hi Ed,

 

Argos will time out on large tables.  You can edit the query properties on the datablock to limit the number of records and/or change the query timeout.  From the MAPS configuration site, you can also change the default query timeout.  

 

In our case, we have over 20 million records in TBRACCD and over 39 million in SFRSTCA.  As long as we are limiting the number of records returned (e.g., limiting by student and/or term), we do not often have query timeouts.

The efficiency of your query depends on a few things... the indexes that are applied to the table , if your parameters are referencing indexed columns, how the query is written, whether you are using production tables vs. mart tables, and how many records are being returned.

 

Unless your dba is ok with changing the indexes on these delivered tables, then you are stuck with what Ellucian provides.

 

I find that using parameters, such as TBRACCD_ACTIVITY_DATE < sysdate, will definitely slow your query down as the activity date field is not indexed in the table.  

 

If you have to make a large query on non-indexed fields, try breaking it up into parts.  For example, I might have a variable that I create that limits my list to certain pidms (one column, pidms only).  So the logic is in this variable and is very lean and does not include other large tables when possible.  Then in my query I make the pidm=variable pidm  so that I'm referencing an indexed field, and the query doesn't have to search the whole table.  Other things that help with efficiency is to reference the table schema and give the table an alias, then in your query use alias.TBRACCD_PIDM -- the more information you can give to your query the better it will behave :).  Avoid cartesian queries -- be sure all your tables are joined on the primary indexes.

 

Like Ace said, the more records you are returning the slower the query.  

 

Using production tables can be very inefficient because they are not normalized (so you have to use more tables to get your data) and you are fighting with others on access to the table, like if they are querying it or are updating it in Banner.  We created a simple and quick data mart that copies production tables nightly (only the ones we need of course) and then we can query on those static tables.  It makes a huge difference.   That was the quick solution, and then from there we started building custom data mart tables specifically for reporting (de-normalized) based off of those copied tables.  The data is refreshed every 24 hours, so it is not live, but we haven't had an issue with it.  It actually helps since users across departments can pull the same report at anytime in the day and get the same data.  If you are not I.T., then this may not be feasible, or it may be a project in the works for future reporting.  If you are using Oracle, adding the records to a temporary table first (global temp tables are awesome), then using the temp table to restrict your output, increases efficiency.  But the configuration manager would have to allow you to do that.

We have similar number of records in TBRACCD and SFRSTCA (20+ mil) and rarely have problems when querying these tables via Argos.

Chamblin hit the nail on the head; TBRACCD especially has fairly useless indices for filtering data beyond pidm and term so others can slow a query down.

 

I find a quick way to cut the number of records returned on these tables is to perform a where exists for a much smaller number of rows* to avoid a full table scan:

 

select *  from tbraccd
                    where tbraccd_activity_Date >= to_date('08/01/2017','mm/dd/yyyy')
                     and exists (select sfrstcr_pidm from sfrstcr where sfrstcr_term_code = :parm_dd_Term and sfrstcr_pidm = tbraccd_pidm)
 

*although I realize SFRSTCR is just as big.
