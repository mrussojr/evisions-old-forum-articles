We do this type of thing frequently. The logic inside the list box or multi-column box utilizes a union statement. For example: 

select distinct sibinst_pidm,
    b.spriden_id,
    b.spriden_last_name||', '||b.spriden_first_name||' '||nvl2(b.spriden_mi,substr(b.spriden_mi,1,1)||'.','') advr
  from sibinst, spriden b
where sibinst_pidm = b.spriden_pidm
and sibinst_advr_ind = 'Y'
and b.spriden_change_ind is null
and (spriden_id = :parm_Eb_advisorID )
and :parm_CB_allAdv = 'N'
union
select null, null, '**all advisors**' from dual
where (:parm_CB_allAdv = 'Y')

It looks like it ought to work as-is. So, I'd recommend reviewing obvious things.... I'd double-check that you are using the correct case on what the checkbox is looking for vs what your SQL is coded for, that you didn't accidentally switch the N and the Y,  and that there are no stray spaces -- 'y' vs 'Y' vs ' Y '. Also, is your listbox set to be Required=Yes? Because it looks like it's not required if someone puts a check in the checkbox so you would not want it to be waiting for for a required value in the listbox. Double-check there's no typo in the parameter name --  :adviseemnu_CB_Alladv vs  :advisemnu_CB_Alladv vs  :adviseemenu_CB_Alladv sort of thing. 

 

If you take out the line entirely, does it work and give you advisors of all types? If you then put back in just "and :adviseemnu_CB_Alladv = 'Y' ", does it still work?
