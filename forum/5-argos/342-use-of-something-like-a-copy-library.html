We have about 100 datablocks that are scheduled to run overnight.  Right now, when there is a change in semesters, we have to go into each datablock and change a parameter to the latest semester.  In addition, we have to also go into the scheduling area and change the schedule start date and schedule end date to match the new semester.

 

does anyone know of a way that we might use in order to make these semester start time and end time easier?  Is there anything like a copy lib, where you only need to make changes in one library, and then that library gets used by multiple programs?  The reason I am looking for this is that after several years of making these multiple mass changes manually, it has become a very cumbersome task and I would love to only have to make these changes once.

There are several possibilities - and one size may not fit all:  Most of our dashboards have the user first select a year, then the terms are automatically limited.

  • if you have a reporting schema of some sort where an ad hoc table can be built and maintained, it could serve as a reference.  You might want to have multiple records to include a hard-coded term, start and end dates for "current term", "next term", "last term" etc. This would allow for datablocks to monitor registration for next term, grades for the last term, enrollment drops for the current term etc....
  • You can reference the SATURN.STVTERM table and/or SOBTERM and SOBPTRM tables and relate the dates found there to SYSDATE in the datablock's SQL code
  • The datablocks can use SQL to determine the term using the structure of your term codes 
        Do an extract year from the SYSDATE and string it with the Fall/Spring/Summer suffix.  
        You can even have it convert the year to a number, add or subtract numbers to get next year or last year and string it together to make a range for terms

Here is an SQL for a drop-down term selection parameter written by a collegue:

SELECT DISTINCT
STVTERM_CODE AS "TERM",
STVTERM_DESC AS "DESCRIPTION"
FROM (
SELECT X.STVTERM_CODE,X.STVTERM_DESC,ROWNUM RNUM
FROM STVTERM X
WHERE TRUNC(SYSDATE-100) <= (X.STVTERM_END_DATE)
AND SUBSTR(STVTERM_CODE,-2,6) <> '99' ---exclude non-credit term
ORDER BY STVTERM_CODE)
WHERE RNUM <=3
ORDER BY STVTERM_CODE

Perhaps you could store values in GTVSDAX to represent reporting terms, then you could select that value from your Argos program.

 

In my example below, the internal group code is ARGOS_REPORT_TERM, so you could have multiple rows representing prior, current, and future terms.  In your Argos program, remove the prompt for term and build the term into the codeblock as a selection from GTVSDAX.

 

GTVSDAX_EXTERNAL_CODE CURRENT_TERM
GTVSDAX_INTERNAL_CODE 201810
GTVSDAX_INTERNAL_CODE_GROUP ARGOS_REPORT_TERM
GTVSDAX_DESC terms for reporting
   
GTVSDAX_EXTERNAL_CODE PRIOR_TERM
GTVSDAX_INTERNAL_CODE 201720
GTVSDAX_INTERNAL_CODE_GROUP ARGOS_REPORT_TERM
GTVSDAX_DESC terms for reporting

(edit...  I like thomas' solution above)

 

Agree.  Any processing that relies on human intervention will eventually fail    I would automate the updates by writing code that determines the semester based on the current date (sysdate), if at all possible -- which is a lovely project in itself which would require updating all of those queries.  

 

In the meantime, there is no library, per se, that would do what you wanted.  The argos library will hold commonly used variables for easy distribution, but you would still have to update them individually in the datablocks.  The only thing that could come close to what you wanted was a custom table built into your data source (Banner PROD for example), that you could update once and it would feed the datablocks.  But even then you are still looking at updating all of the query code to accommodate the new table.  

 

Sigh, I don't see any easy way out for you
