Hello -

 

I suspect this has come up before and perhaps it was answered but currently we don't know a good way to handle things like gurmail updates if letters are run from Argos. We might want to schedule a job each day to send out letters from gurmail that haven't let been printed. The trouble is that the job can run but the gurmail table can't be updated. With no automated way to update these print dates when a job runs, it is really cumbersome to have users have to go back in each day and run some other custom job to update print dates.

 

Is there a way schools are handling this so we can actually leverage the letter generating capabilities within Argos? Evisions has stated that Argos can be a replacement for tools like SQR but with no way to handle the updates without having to go back and do something else, it doesn't feel like a good replacement for all of the insert, update, and delete processing that we can do with SQR as records are processed.

 

Any tricks or tips appreciated.

I have never worked with this specific problem, but my first inclination is:

 

I believe that a connection has to be set up properly in the MAPS configuration to allow for updates and script running. If you create a duplicate connection as the one you are using to query, you can run a variable in the datablock with that second connection that runs an update statement on the table. For example, you can tie that variable to only execute when a checkbox is checked on the dashboard. After your schedule executes the job you currently do, add another task at the bottom that executes the report again but with that checkbox checked.

 

That might work somehow? That is definitely where I would start if I was trying to solve this problem.
