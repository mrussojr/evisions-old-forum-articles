We have a request to generate letters for students that will indicate the courses they are registered for and tuition that they owe, but to only have the letters go out once.  Our problem is that our systems group does not want us updating any tables, such as the GOREMAL table where we could put a flag that would let us know that the student has already received a letter. 

If anyone has a suggestion as to how we can issue letters with this update restriction on us, we would truly appreciate it.

They may be more agreeable if you offer to use the GURMAIL table instead... which is where you would record these types of letter distributions.  GOREMAL is too valuable to campus operations that they are probably only concerned about security for this particular object.  If that doesn't work, you may have to rely on a date stored somewhere that would trigger a letter.  For example, the registration table SATURN.SFRSTCR has an activity date you could use to say, send letters to everyone who signed up for a particular course in the past week.  Or you could use the accounts receivable posting date of the tuition transaction(s) to trigger a letter.  

They could create a specific table just for you all that you can update all you wanted without hurting anything else.  As for me, I usually create a stored procedure on the server that manages the DDL statements and grant execute access to the user -- this way I control the code and they can have access to the procedure but not to the tables themselves.  Users can pass in their variables in the query with an execute statement.  

 

AND here's another possible option, you can record the letter pidms and letter date, etc, by pasting them into a spreadsheet on a drive that argos can read and write to.  once the letters are finished, you could have argos update the spreadsheet instead of a table and then re-use the file to exclude prior letter recipients.  You'd have to create an ADO connection to the file by having a configuration admin create a data source to the file.  And since it is a different data source than your Banner source, you'd use a variable in the datablock that pointed to the database file and plucked out the necessary pidms.  You can also create a multi-column or list box that points to the file data source which will list its contents for manipulation in other parts of the Banner datablock.

We don't have The banner communications module set up at all at our institution, due to the fact that when we implemented banner we only put user names (not formatted e-mails) into GOREMAL.  When run a letter generation process for the FinAid Awards, and populate a date and code in GURMAIL.  We then have an Access Database (There's a project to port it to Argos on my task list) that generates the letter based on GURMAIL Date and Code.  

 

  I like the Saturn.SFRSTCR option - your datablock  could have a radio button to let you choose between "All Students" and "Students with Registration Activity after XXXX" so you could do a broad initial e-mail, and then follow up with those after.

 

  Also - Similar to Chamblin's idea, you could talk to your systems team about getting access to a POPSEL to store the PIDMS of people who already have a letter sent for the given term, and filter PIDMs from there.  Since POPSELs are more self contained than the GURMAIL table, you run less of a risk of ruining things when you mess with them.
