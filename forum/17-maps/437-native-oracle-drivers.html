Yes, with 12c but in a limited capacity.  Specifically, we installed the Oracle native driver on our sandbox MAP server and did some limited testing with it using a non-production 12c database (Oracle 12c EE, Release 12.1.0.2.0).  I noted a 25% reduction in query time in one case and so far we have not encountered any issues using the native driver.

 

However, native driver support  for Oracle and MS SQL Server was part of the MAPS 5.1 release.  There have been several bugs reported since then and there  could be more that just haven't been discovered yet.  We tend to know about them only if we reported the bug or see it has been resolved in a new release.

 

When I searched all the release notes for MAPS I found the following items related to native drivers:

 

5.4 Data connections Selecting the entire contents of a multi-column list box was taking significantly longer when using a native database connection than it did when using an ADO connection. MAPS-1472

Resolved in MAPS 5.4

 

5.2.1 Connections Attempting to upgrade the IntelleCheck database using a native Oracle connection resulted in missing records in the evibcmac table, requiring a reload of .icc files. MAPS-1338

Resolved in MAPS 5.2.1

 

5.2.1 Connections When using the native Oracle database driver, queries that selected identical fields without using aliases would fail if the maximum number of records to read was set to something other than "no max". MAPS-1288

Resolved in MAPS 5.2.1

 

5.2 Connections When using the native SQL Server database driver, the functions of the !> (not greater than) and !< (not less than) operators were reversed. MAPS-1226

Resolved in MAPS 5.2

 

5.2 Connections When using the native Oracle database driver, the Service Name property was not picking up the list of configured Oracle databases from the tnsnames.ora file. MAPS-1229

 

Resolved in MAPS 5.2

 

5.2 Connections When using the native Oracle database driver, Unicode characters were being converted to their ANSI equivalents when executing queries. This resulted in the query returning data with missing characters, such as those required for Latin, French, Arabic, Chinese, etc. MAPS-1240

Resolved in MAPS 5.2

 

5.2 Connections When using the native SQL Server database driver, using certain reserved characters ('!', '&', ':', '?', '{', and '}') in a query would cause the SQL to be interpreted incorrectly. MAPS-1308

Resolved in MAPS 5.2

 

5.2 Connections When entering the password for a native database connection on the Driver Properties screen, the password characters were not being obscured on the screen. MAPS-1257

Resolved in MAPS 5.2

 

 

When I searched all the release notes for Argos I found the following items related to native drivers:

 

5.4 Data connections When using a native database connection, attempting to use numbers stored in string format as part of a SUM function within an extract report caused the report to fail with the error "Expected a numeric value on line 1 position 6". AR-4329

Resolved in Argos 5.4

Our DBA took care of the database upgrades (ODAC, homes, etc.) so I don't have any detailed feedback for you in that area.   

 

Our sandbox instance of MAPS was cloned from a production instance about a year ago.  Is it recommended to use a cloned version of your production MAP server?  That depends on your tolerance for pain. 

 

If during your testing you were discover a new critical defect with the native driver (meaning it could crash the MAP server) then you would probably be very happy if you were using a non-production instance when you discovered the defect.
