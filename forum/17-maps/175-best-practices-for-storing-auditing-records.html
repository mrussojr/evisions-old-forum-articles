Auditing records are also available "offline" if you have enabled "Archive audit detail records before purging the records from the database".  With this archive process, the auditing records are saved as XML with no data loss.  However, I did come across an issue where 2 of the 3 dates in the detail file are offset by minus 1 month.  The 2 incorrect dates can be fixed using the 1 correct date.

 

So, if you're worried about performance/storage on your production system, you could set the Archive process to write to cheaper/slower storage accessible by the server and bump up the "Number of audit archive files to keep" to a number that equals one year.  In our case, the archive process creates 2 files per day.  So if we wanted a full year of auditing records in the archive we'd need to set the "Number of audit archive files to keep" to 730.

Yes, you're probably looking at using Access or Excel to query the archived XML auditing records.  If you don't need to report usage data often then it's probably a viable solution.

 

Like you said, the alternative is to increase the retention of audit records in the Data Warehouse.  You'll "pay" for this option by needing to dedicate more disk space to the MAPS server.  What I don't know is how well SQLite performance scales with database (file) size.  My gut tells me it would probably be ok to increase retention in the Data Warehouse to one years worth of auditing records since it will be primarily Read operations by what I assume are only a handful of users.
