Hi Kevin,

Technically, I don't monitor the MAPS Service directly.  I do monitor what HTTP status code is returned by the eLauncher page which is served up MAPS' built in lightweight HTTP/S server.

 

I interrogate the eLauncher pages of our production and development instances every fifteen minutes using a PowerShell script.  The script loops through a text file containing the list URLs I want to check.  For each URL the HTTP status code is checked and anything that doesn't start with a '2' (https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) I classify as an error and send an alert email to me and my fellow administrators.  This process runs from 7AM to 7PM to cover our typical hours of user activity.

 

So far so good.  It has already alerted us to problems before our users could submit a helpdesk ticket.  One deficiency is that I have the script running via Task Scheduler on my own workstation so I need to leave it powered on 24/7.  The first and only issue I had with this configuration was last week when the carpet cleaning company accidentally turned off my powerstrip .  It's now on my agenda to move it over to a genuine server.

 

That being said, the PowerShell script only checks if the eLauncher page is responsive.  In order to prove that the core scheduling and querying services are functional I have two special scheduled reports that I call "pulse checks".  One scheduled report uses the MAPS Database connector to verify that at least one of the underlying SQLite databases (MAPS Database) is functional.  The second scheduled report uses a Banner Production connector to verify that MAPS can communicate with our Banner production database -- this second scheduled report is relatively new because we recently learned that certain poorly written queries can cause individual connectors to become unresponsive while other connectors remain unaffected.  These two scheduled reports are sent every hour as proof that the core scheduling and querying services are functional.  If I don't receive an email on the hour then I know there is a potential problem.  I stress potential problem because it's easy to forget that MAPS is simply handing off emails to an SMTP server and sometimes things break there or further downstream.

Just to add to what Ace Gayhart said about load balancing...

 

We revamped all our servers about a year ago, including the ones hosting MAPS.  I read somewhere in the docs that the servers could be load balanced so we went ahead with this configuration, 2 servers under one load balancer.  Everything got setup and bother servers independently tested then we connected them to the load balancer.  While the docs were technically correct in that they can be load balanced, they really can't and really shouldn't.

 

One gleaming defect in how MAPS is designed: there is no central database (it can't be a clustered configuration).  I don't know why this didn't strike me before going through this exercise, but we wasted a fair amount of time just to realize the software isn't capable of clustering and therefor isn't capable of true load balancing (though we still run our single server through a load balancer for consistency).  Basically, any configuration changes (Argos, FormFusion, IntelleCheck, MAPS, etc.) made to one server aren't copied over to the other as they use local instances of a SQLite database.  We discussed having all DataBlock designers only login to server1 and make changes and we could copy the nightly backup to server2 the following morning but this type of thinking can get pretty messy, pretty quickly.  Not to mention scheduling... both servers would be running the same report schedules and everything would be duplicated.

 

My 2¢: don't even bother with load balancing until MAPS can store the configuration in a central database and not a SQLite database located inside the installed folder, at least not for failover and true load balancing.  It also needs to be able to handle scheduling while multiple instances are running.  If multiple servers could pull from a separate, external database the configuration issue would be resolved.  If multiple servers could cross-talk and only one handle schedules at a time it would handle scheduling.
